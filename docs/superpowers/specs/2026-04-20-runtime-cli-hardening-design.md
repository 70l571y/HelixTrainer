# Runtime CLI Hardening Design

**Scope:** первый пакет улучшений для `HelixTrainer`, направленный на
исправление хрупких мест рантайма и CLI без расширения продуктового
набора команд.

## Goal

Сделать текущий CLI-поток надёжнее и понятнее для пользователя:

- исправить нерабочую проверку обновлений;
- привести путь хранения данных к платформенным стандартам;
- убрать хрупкий post-challenge flow в `play`;
- улучшить диагностику ошибок при запуске Helix;
- перестать молча игнорировать важные ошибки файлового и DB IO.

Этот пакет не включает новые продуктовые команды вроде `doctor`,
`completion`, `--json`, новые фильтры или переработку progression.

## Non-Goals

- не добавлять новые challenge-ы или новые пользовательские режимы;
- не переписывать весь CLI на новую архитектуру;
- не вводить raw terminal input, если это потребует отдельного слоя TTY;
- не делать полноценный auto-update installer.

## Current Problems

### 1. Upgrade flow фактически не работает

Текущий код строит URL к несуществующему `main/go/go.mod` и затем ищет
в `go.mod` комментарий версии, которого нет. В результате команда
`hxtrainer upgrade` не может надёжно определить latest version.

### 2. Config path не соответствует platform conventions

Текущий код вручную маппит OS на каталоги и трактует macOS как Linux/XDG.
Это расходится с поведением стандартной библиотеки Go и приводит к
неожиданному пути для пользователей macOS.

### 3. Post-challenge interaction вводит пользователя в заблуждение

После завершения challenge интерфейс пишет про `j | k | esc/q` как будто
ждёт одиночное нажатие клавиши, но фактически используется строковый ввод
через `Scanln`. Дополнительно переход к следующему challenge сделан через
рекурсивный вызов `runPlay`.

### 4. Ошибки editor launch скрыты

Слой `editor` возвращает только `bool`, поэтому вызывающий код не может
сообщить пользователю, что именно сломалось: отсутствие `hx`, ошибка
завершения процесса, неправильный файл или другая проблема окружения.

### 5. Часть ошибок теряется

Некоторые операции файлового копирования, логирования попыток и похожие
IO-paths либо игнорируют ошибки полностью, либо не различают fatal и
non-fatal случаи.

## Proposed Approach

Рекомендованный подход: точечный hardening без смены общей модели
приложения.

### A. Config path через standard library

`cfg` должен опираться на `os.UserConfigDir()` как основной источник
каталога конфигурации, добавляя подпапку `hxtrainer`. Если стандартная
библиотека не может определить путь, допустим локальный fallback на
текущий упрощённый механизм через home directory.

Это даёт:

- корректный путь на macOS;
- сохранение XDG/Windows semantics там, где они уже ожидаемы;
- меньше ручной OS-specific логики.

### B. Editor API с явной ошибкой

`editor.OpenEditor` должен возвращать `error`, а не `bool`.

Ожидаемое поведение:

- если целевой файл не существует, вернуть filesystem error;
- если `hx` отсутствует, вернуть явную ошибку окружения;
- если `hx` завершился неуспешно, вернуть ошибку процесса;
- вызывающий код должен напечатать понятное сообщение с причиной.

### C. Один play loop без рекурсии

`runPlay` должен работать через явный цикл с состоянием:

- выбрать challenge;
- подготовить workspace;
- открыть editor;
- провалидировать результат;
- спросить действие пользователя;
- либо повторить тот же challenge, либо перейти к следующему, либо выйти.

Рекурсивный переход к следующему challenge должен быть удалён.

### D. Честный post-challenge prompt

В этом пакете не вводим raw key capture. Значит UI должен соответствовать
реальному вводу.

Ожидаемое поведение:

- prompt явно просит ввести `j`, `k` или `q` и нажать Enter;
- `esc` не рекламируется как supported path, пока оно не реализовано
  по-настоящему;
- невалидный ввод обрабатывается повторным приглашением.

### E. Надёжный upgrade behavior

Команда `upgrade` не должна притворяться fully automatic update, если она
только подсказывает команду пользователю.

В рамках этого пакета:

- latest version должна определяться из надёжного внешнего источника;
- current version берётся из `buildinfo`;
- если latest определить не удалось, пользователю показывается честная
  ошибка проверки обновлений;
- если newer version найдена, команда показывает корректную инструкцию
  обновления через `go install ...@latest`.

Предпочтительный источник latest version: GitHub Releases API или tags API
репозитория `70l571y/HelixTrainer`, без парсинга `go.mod`.

### F. Явная обработка non-fatal и fatal errors

Для всех изменяемых участков нужно разделить ошибки на два класса:

- `fatal`: нельзя продолжать текущий workflow;
- `non-fatal`: можно продолжать, но пользователь должен увидеть warning.

Примеры:

- ошибка записи temp file: fatal;
- ошибка `copyFile` для обязательного extra file: fatal;
- ошибка логирования попытки после завершения challenge: non-fatal warning,
  но сам результат challenge уже считается показанным пользователю.

## Components And File Boundaries

### `internal/cfg/cfg.go`

Ответственность:

- вычисление user config directory;
- формирование путей к DB и встроенным challenge data.

Планируемое изменение:

- перейти на `os.UserConfigDir()`;
- сохранить маленький fallback path только на случай ошибки stdlib.

### `internal/editor/editor.go`

Ответственность:

- запуск `hx` в нужной рабочей директории;
- проверка наличия редактора.

Планируемое изменение:

- заменить `bool` на `error`;
- улучшить диагностику причин сбоя;
- при необходимости сохранить `HelixInstalled()` только если она ещё нужна.

### `internal/app/app.go`

Ответственность:

- orchestration команд `play`, `stats`, `list`, `upgrade`;
- основной interaction flow.

Планируемое изменение:

- переписать post-challenge loop без рекурсии;
- скорректировать prompt под line-based input;
- перестать глотать важные ошибки;
- переписать `upgrade` на новый источник latest version.

### `README.md`

Ответственность:

- документировать реальный путь хранения данных;
- описывать обновление без ложных обещаний.

Планируемое изменение:

- обновить раздел о расположении данных;
- при необходимости уточнить поведение `upgrade` и/или рекомендованный
  способ обновления.

## Data Flow

### Play flow

1. Команда `play` загружает challenge list.
2. Выбирается конкретный challenge:
   либо явно по ID, либо smart selection.
3. Создаётся workspace и запускается `hx`.
4. После завершения editor session результат валидируется.
5. Попытка логируется; ошибка логирования показывается как warning.
6. Пользователь вводит следующее действие строкой:
   `j` = next, `k` = retry, `q` = quit.
7. Цикл продолжается без рекурсивного self-call.

### Upgrade flow

1. Команда получает текущую версию из build info.
2. Команда запрашивает latest release/tag через GitHub endpoint.
3. Версии сравниваются как release strings.
4. Пользователь получает одно из сообщений:
   already latest / update available / check failed.

## Error Handling

### Play command

- если `hx` не найден, показать действие:
  “установите Helix и убедитесь, что `hx` доступен в PATH”;
- если editor process завершился с ошибкой, показать саму причину;
- если дополнительный файл не удалось скопировать, прекратить запуск
  challenge с ошибкой;
- если не удалось залогировать попытку, вывести warning после результата.

### Upgrade command

- network/API ошибка должна давать явное сообщение о невозможности
  проверить обновления;
- malformed version response не должна приводить к ложному “up to date”.

## Testing Strategy

Нужны автоматические тесты на следующие сценарии:

- config dir использует platform-aware base path;
- `editor.OpenEditor` возвращает диагностируемые ошибки;
- post-challenge action parsing работает без рекурсии;
- `upgrade` корректно обрабатывает:
  latest == current,
  latest > current,
  network/parse failure.

Если часть логики сложно тестируется в текущей форме, допустимо вынести
небольшие helper functions для выбора action и интерпретации version check.

## Risks And Tradeoffs

### Не вводим raw terminal mode сейчас

Это сознательный компромисс. Реальный raw key capture улучшил бы UX, но
вынес бы пакет в отдельную TTY-задачу с более высоким риском и большим
объёмом тестирования.

### Upgrade остаётся advisory, а не auto-installer

Это тоже сознательно. Для Go CLI безопаснее и проще честно сообщать о
новой версии и команде обновления, чем пытаться управлять установкой за
пользователя.

## Success Criteria

Пакет считается завершённым, если:

- `hxtrainer upgrade` больше не зависит от фиктивного парсинга `go.mod`;
- путь хранения данных соответствует стандартам Go platform dirs;
- `play` больше не использует рекурсивный переход на следующий challenge;
- prompt после challenge соответствует фактическому способу ввода;
- пользователь видит конкретную причину ошибок запуска Helix;
- критичные IO-ошибки больше не теряются молча;
- README отражает новое реальное поведение.
