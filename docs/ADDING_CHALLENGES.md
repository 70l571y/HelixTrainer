# Добавление новых челленджей в HelixTrainer

Это руководство описывает процесс создания новых челленджей для HelixTrainer. Оно предназначено для разработчиков и AI-агентов для обеспечения согласованности и качества во всём наборе челленджей.

**ВАЖНО ДЛЯ AI-АГЕНТОВ:**
Если вы обнаружите новые паттерны, лучшие практики или ограничения (например, "избегайте выделения %"), вы **ДОЛЖНЫ** обновить этот файл, чтобы отразить полученные знания. Поддерживайте этот документ актуальным.

## 📂 Структура каталогов

Челленджи организованы по языкам программирования в каталоге `challenges_data`.

```
challenges_data/
├── go/
│   ├── challenge_id_1/
│   │   ├── config.json
│   │   ├── start.go
│   │   └── goal.go
│   └── challenge_id_2/
│       └── ...
├── python/
│   └── ...
└── ...
```

## 📝 Необходимые файлы

Минимальная папка челленджа содержит три файла:

1.  **`config.json`**: Метаданные и конфигурация.
2.  **`start.ext`** (например, `start.go`): Начальное состояние основного файла.
3.  **`goal.ext`** (например, `goal.go`): Целевое состояние основного файла.

Для более сильных workflow-задач папка **может** содержать дополнительные
файлы. Это особенно полезно для:

*   multi-file refactor
*   LSP rename / references
*   diagnostics и workspace picker сценариев
*   split / buffer / navigation challenge-ов

### 1. `config.json`

Это управляющий файл. Он должен быть валичным JSON со следующей структурой:

```json
{
    "id": "unique_snake_case_id",
    "title": "Читаемое название",
    "description": "Краткое (1 предложение) описание задачи.",
    "difficulty": "Easy" | "Medium" | "Hard",
    "language": "go",
    "judge_mode": "go_ast" | "exact" | "ignore_whitespace",
    "start_file": "start.go",
    "main_file_name": "challenge.go",
    "goal_file": "goal.go",
    "goal_files": ["goal.go"],
    "extra_files": ["service.go", "report.go"],
    "validation": {
        "challenge.go": "goal.go",
        "service.go": "goal_service.go"
    },
    "git_dirty_files": {
        "helper.go": "dirty_helper.go"
    },
    "tips": "Пошаговые подсказки с командами Helix.\nДержите строки < 80 символов.\nИспользуйте \\n для новых строк.",
    "tags": [
        "tag_из_helix_labels_md",
        "another_tag"
    ],
    "author_time": 10.5
}
```

#### Детали полей:

*   **`id`**: Уникальный идентификатор, snake_case (например, `extract_method`). Должен совпадать с именем папки.
*   **`title`**: Отображается в меню (например, "Извлечь метод").
*   **`description`**: Краткая цель. **ДОЛЖНО** указывать, *что* нужно изменить (например, "Измените 100 на 200"). Не полагайтесь на подсказки для передачи цели.
*   **`difficulty`**: Субъективный уровень сложности.
*   **`language`**: ID языка программирования (используется для структуры папок и подсветки синтаксиса).
*   **`judge_mode`**:
    *   `"go_ast"`: (Рекомендуется для Go) Парсит код в AST Go. Игнорирует различия в пробелах/комментариях.
    *   `"exact"`: Посимвольное сравнение. Самый сложный режим.
    *   `"ignore_whitespace"`: Удаляет пробелы перед сравнением. Хорошо для манипуляций с текстом.
*   **`goal_file`**: Основной goal-файл для single-file сценария.
*   **`main_file_name`**: Необязательное имя основного файла во временной
    директории challenge-а. По умолчанию рантайм создаёт `challenge.go`
    (или файл с тем же расширением, что у `start_file`). Используйте это
    поле для сценариев, где имя основного файла само является частью
    workflow-а или требуется для более правдоподобного multi-file layout,
    например `matrix_processor.go`.
*   **`goal_files`**: Альтернативные goal-файлы для одного и того же
    основного буфера, если вы хотите разрешить несколько корректных
    состояний.
*   **`extra_files`**: Дополнительные файлы, которые будут скопированы во
    временную директорию challenge-а и доступны из Helix.
*   **`validation`**: Явная карта валидации multi-file сценария.
    Ключ - имя файла во временной директории, значение - goal-файл внутри
    challenge directory. Используйте этот режим, если важны изменения
    сразу в нескольких файлах.
*   **`git_dirty_files`**: Необязательная карта для challenge-ов,
    которым нужен dirty git workspace с самого старта. Ключ - имя файла
    во временной директории, значение - fixture-файл внутри challenge
    directory, который будет записан поверх committed baseline после
    `git init` и первого commit. Это полезно для `changed file picker`
    сценариев.
*   **`tips`**: Строка, содержащая подсказки.
    *   **Форматирование**: Используйте `\n` для разрывов строк.
    *   **Ограничение**: Держите строки **< 80 символов** для предотвращения проблем с переносом терминала.
    *   **Содержание**: Явно упоминайте клавиши Helix. Обучайте наиболее эффективному/идиоматичному методу (например, используйте `mr` для замены скобок, а не удаление+вставка).
    *   **Официальные no-yank команды**: для modern Helix prefer
        documented shortcuts `Alt-d` и `Alt-c`, если задача тренирует
        delete/change without yank как действие. Нотацию `"_d` и `"_c`
        используйте там, где challenge именно про registers и blackhole
        register, а не просто про “не затирать yank”.
    *   **Для core hotkey challenge-ов**: если у Helix есть официальный
        default binding, пишите его в tips явно. Не заменяйте конкретный
        хоткей расплывчатым “через hotkey-driven путь”.
    *   **Если официального default binding нет**: либо оставляйте задачу
        в optional command-line track, либо проектируйте её вокруг
        соседнего workflow с documented hotkeys.
*   **`tags`**: Список тегов. **ДОЛЖНЫ** браться из `HelixLabels.md`.
*   **`author_time`**: Целевое время для получения вехи "Author" в секундах.

### 2. `start.go`, `goal.go` и дополнительные файлы

*   **Только содержимое**: **НЕ** включайте заголовок с описанием челленджа (например, `# Task: ...`) или футер с подсказками. Игровой движок автоматически внедряет их во время игры для обеспечения согласованного форматирования.
*   **Чистый код**: Убедитесь, что `start`, `goal` и все goal-файлы из
    `validation` валидны и парсятся, если используете `go_ast`.
*   **Минимальная разница**: Старайтесь держать файлы `start` и `goal`
    достаточно похожими, чтобы разница была читаемой, но достаточно
    разными, чтобы требовать конкретных действий Helix.
*   **Имена файлов**: Если challenge multi-file, выбирайте имена, которые
    усиливают нужный workflow. `service.go`, `report.go`, `worker.go`
    учат ориентироваться в проекте лучше, чем абстрактные `a.go`, `b.go`.

## 🧩 Single-File vs Multi-File

Используйте single-file challenge, если навык живёт в пределах одного
буфера:

*   movement
*   selection
*   surround
*   registers
*   локальные text-object и search/replace сценарии

Используйте multi-file challenge, если пользователь должен освоить
именно workspace-мышление:

*   LSP references или rename через несколько файлов
*   diagnostics picker с переходом между файлами
*   global search -> jump -> edit в другом файле
*   split / buffer / close-others workflow

Практическое правило:

*   если задачу можно честно решить, не открывая второй файл, скорее всего
    single-file достаточно
*   если сила сценария в навигации, координации буферов или LSP по
    проекту, делайте multi-file

## ✅ Принципы качества

Создание высококачественных челленджей обеспечивает эффективное обучение пользователей. Следуйте этим принципам:

1.  **Реализм**:
    *   Код должен выглядеть как настоящий код. Определяйте переменные перед использованием (например, `a := 1`, `b := 2` перед `L := []int{a, b}`).
    *   Избегайте бессмысленных сниппетов вроде `foo bar baz`, если это не строго необходимо для задачи манипуляции текстом.
    *   Функции должны иметь тела, даже если это просто `return` или print-вызовы.

2.  **Соответствующий контекст**:
    *   **Длина**: Для челленджей перемещения (например, `movement_long_jump`) убедитесь, что файл достаточно длинный (50+ строк), чтобы прыжок был полезным.
    *   **Структура**: Для выделения объектов (например, `maf`) сделайте блок достаточно большим, чтобы ручное выделение было утомительным.

3.  **Синтаксическая валидность**:
    *   Убедитесь, что `start`, `goal` и все файлы из `validation`
        являются валидным, парсируемым кодом.
    *   Это критично для правильной работы `judge_mode: "go_ast"`.

4.  **Обоснование**:
    *   Задача должна демонстрировать, *почему* функция Helix полезна.
    *   Пример: Не просите удалить 3 строки для обучения `maf`. Попросите удалить функцию из 20 строк.
    *   Для workflow-задач причина должна быть ещё сильнее: пользователь
        должен почувствовать, почему Helix удобен именно на проекте, а не
        только в одном буфере.

5.  **Разделение инструкций и подсказок**:
    *   **Описание**: Должно говорить пользователю, *что* достичь (например, "Оберните выражение в скобки").
    *   **Подсказки**: Должны говорить пользователю, *как* достичь этого (например, "Используйте 'ms('").
    *   Не скрывайте цель внутри подсказок.

6.  **Идиоматичный Helix**:
    *   Обучайте "способу Helix". Если задача может быть выполнена с помощью `mr` (match replace), не обучайте `d` затем `i`.
    *   Если подсказка использует конкретную функцию (например, `s` для regex-выделения), убедитесь, что соответствующий тег (`select_regex`) присутствует в `config.json`.
    *   Если в official keymap уже есть dedicated hotkey для действия,
        предпочитайте его прямое обучение. Примеры: `Alt-d` вместо
        register workaround для no-yank delete, `<space>R` для clipboard
        replace, `Ctrl-s` / `Ctrl-v` в picker для split-open.

7.  **Стратегия выделения**:
    *   **Избегайте `%`**: В игровой среде файл включает внедрённый заголовок и футер с подсказками. Использование `%` выделяет эти артефакты, что ломает многие команды трансформации.
    *   **Используйте специфичные селекторы**: Обучайте `vip` (выделить внутренний параграф), `x` (выделить строку) или `mi...` (выделить внутри объекта) для нацеливания конкретно на блок кода пользователя.

8.  **Надёжность и анти-читерство**:
    *   **Разные значения**: При просьбе изменить несколько вхождений убедитесь, что значения различаются (например, `timeout=30`, `timeout=6000`). Это предотвращает простые решения через Find/Replace (`%s`) или макросы и заставляет использовать структурную навигацию.
    *   **Смешанные типы**: Используйте комбинации списков, кортежей и мап. Это предотвращает "накрутку" челленджа поиском одного закрывающего разделителя (например, `)`) для нахождения конца блока.
    *   **Шум**: Добавьте "шумный" контент (например, строки, содержащие скобки) для сломать простой regex-поиск, если цель — структурная навигация (`mm`).
    *   **Multi-file устойчивость**: Если цель в том, чтобы пользователь
        действительно перешёл в другой файл, валидируйте и второй файл
        тоже, а не только основной буфер.

9.  **Workflow-first проектирование**:
    *   Сначала формулируйте рабочий сценарий, потом подбирайте команды.
    *   Хороший вопрос для автора: “Какой кусок реальной работы по коду
        имитирует этот challenge?”
    *   Если challenge отвечает только на вопрос “как потренировать кнопку”,
        он слабее, чем challenge, который учит замкнутому циклу
        `find -> select -> transform -> verify`.
    *   Для core-track-а предпочитайте hotkey-first сценарии. Если тот же
        навык можно правдоподобно тренировать без `:command`, выбирайте
        именно hotkey path.

10. **Anti-cheese проверка**:
    *   Перед добавлением challenge-а задайте вопрос: “Можно ли решить
        это быстрее тупым insert/delete путём, вообще не тренируя нужный
        workflow?”
    *   Если да, добавьте структуру, шум, второй файл или selective
        validation так, чтобы нужный приём стал самым естественным.
    *   Хороший core challenge должен подталкивать к intended workflow,
        а не только формально допускать его.
    *   Для hotkey-first задач избегайте подсказок и формулировок,
        которые толкают пользователя в command-line, если он не нужен.
    *   Но не делайте tips слишком нейтральными: core challenge должен
        не только запрещать плохой путь, но и подсказывать правильный
        muscle-memory path через конкретные официальные хоткеи.

## ⚠️ Распространённые ошибки

*   **Режим `exact`**: Избегайте этого, если пробелы не являются строго частью челленджа. Это расстраивает пользователей, когда завершающий перевод строки ломает решение. Используйте `go_ast` (Go) или `ignore_whitespace` (Текст) вместо этого.
*   **Расплывчатые инструкции**: "Исправьте код" — плохо. "Измените имя функции на 'process'" — хорошо.
*   **Подавляющие подсказки**: Не вываливайте 5 различных способов сделать это. Обучайте *лучшему* способу.
*   **Короткие файлы**: Helix сияет в больших файлах. Однострочники не демонстрируют мощь `gw` или `mip`.
*   **Однородные данные**: "Измените все `foo` на `bar`" часто лучше решается через `%s/foo/bar`. "Удалите 2-й аргумент из этих 3 различных вызовов функций" заставляет перемещать курсор и делать специфичные правки.
*   **Ложный multi-file**: Не добавляйте второй файл только ради галочки.
    Он должен быть нужен для navigation, validation или orchestration.
*   **Случайный command-line drift**: Если challenge должен тренировать
    hotkeys, не пишите tips в стиле `:o`, `:vs`, `:buffer-next`, если
    тот же сценарий можно выполнить через picker/window/buffer hotkeys.
*   **Реестр тегов отстал от challenge-ов**: если вы добавили новый
    challenge-tag в `config.json`, но не описали его в
    `docs/HelixLabels.md`, считайте это ошибкой набора данных.

## 🗺️ Покрытие функций

Базовый набор меток из `docs/HelixLabels.md` должен иметь хотя бы один
челлендж на каждую функцию Helix. Если вы добавляете новую метку в реестр,
добавьте и хотя бы один челлендж, который обучает её как основному приёму.

Предпочтение отдавайте не количеству, а качеству покрытия:

1.  Добавляйте **dedicated** челлендж, если функция пока встречается только как
    вспомогательный шаг внутри более крупной задачи.
2.  Добавляйте **вариант**, если базовый сценарий уже есть, но не покрывает
    важный реальный паттерн использования.
3.  Обновляйте теги существующих задач, если challenge уже учит нужной
    команде, но метка в `config.json` отсутствует.

Когда tutorial-покрытие уже есть, следующий шаг - не новый label, а
новый уровень владения тем же навыком:

1.  isolated command
2.  applied single-file workflow
3.  multi-step workflow
4.  workspace-aware workflow

См. также
[Helix Learning Path](./LEARNING_PATH.md)
для рекомендуемого порядка такого углубления.

## 🏷️ Теги и категории

Мы используем стандартизированный список тегов для категоризации челленджей по тестируемым функциям Helix.

*   **Источник истины**: См. `docs/HelixLabels.md` в корне проекта.
*   **Выбор**: Всегда проверяйте теги против этого файла. Не придумывайте новые теги без обновления `docs/HelixLabels.md` сначала.

**Распространённые теги:**
*   `movement_basic`, `movement_word`, `movement_find`, `movement_goto`
*   `edit_insert`, `edit_delete`, `edit_change`, `edit_replace`, `edit_case`
*   `select_line`, `select_regex`, `select_split_regex`, `select_split_lines`
*   `selection_reset`, `selection_align`, `select_object`, `select_syntax`
*   `search_basic`, `search_selection`, `register`, `multicursor`

## 📚 Ресурсы

*   **`docs/HelixLabels.md`**: (Корень проекта) Официальный реестр тегов. Используйте это для маркировки ваших челленджей.
*   **Файлы `tutor`**: Ищите `tutor` или `helixTutor.txt` в репозитории для вдохновения в последовательностях обучения.
*   **`docs/LEARNING_PATH.md`**: Используйте как фильтр при выборе новых
    challenge-ов. Он помогает понять, усиливает ли идея обучение
    реальному workflow.

## 🚀 Рабочий процесс для агентов

1.  **План**: Определите функцию Helix для обучения (например, "Несколько курсоров").
2.  **Консультация и обновление меток**:
    *   Проверьте `docs/HelixLabels.md` для правильного тега (например, `multicursor`).
    *   **Критично**: Если функция (например, "Comment" `Ctrl-c`) отсутствует, вы **ДОЛЖНЫ** добавить её в `docs/HelixLabels.md` сначала. Не используйте некорректный тег и не придумывайте его без документации.
3.  **Черновик**: Создайте код `start` и `goal`.
    *   *Совет*: Для `multicursor` убедитесь, что в `start.go` есть выровненные паттерны, которые делают вертикальное выделение или regex-выделение очевидным.
4.  **Конфигурация**: Создайте `config.json`.
    *   Установите `judge_mode` соответствующим образом (обычно `go_ast` для Go).
    *   Напишите `tips`, которые направляют пользователя через конкретную последовательность клавиш (например, "1. Выделите строки. 2. Нажмите 's'. ...").
    *   Если задача multi-file, заранее решите, какие файлы идут в
        `extra_files`, а какие обязаны пройти через `validation`.
5.  **Проверка**:
    *   Строки < 80 символов?
    *   Заголовки удалены из файлов `start`/`goal`?
    *   Теги валидны?
    *   Подсказка избегает выделения всего файла с `%`?
    *   Если challenge multi-file, точно ли второй файл действительно нужен
        для прохождения, а не лежит в папке без функции?
